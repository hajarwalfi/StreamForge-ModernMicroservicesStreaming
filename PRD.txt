Brief : Streaming Vidéo en Microservices avec API REST
Description :
Application de streaming vidéo backend composée de microservices avec API REST testable via
Postman
Contexte :
Une plateforme de streaming vidéo souhaite moderniser son infrastructure IT en développant
une application backend basée sur une architecture microservices. Le système doit gérer les
utilisateurs, les contenus vidéo, les watchlists et l’historique de visionnage à travers des API REST.
Architecture Microservices :
1. video-service : Gestion des contenus vidéo
2. user-service : Gestion des utilisateurs, watchlist et historique
3. gateway-service (proxy) : Point d’entrée unique API (Spring Cloud Gateway)
4. discovery-service : Service discovery (Eureka)
5. config-service : Configuration centralisée. Ce microservice va devoir se connecter à un
repository Git (qui peut être local ou distant) où sont stockés les fichiers de configuration
au format YAML ou Properties
Entités Principales
User Service :
- User : id, username, email, password
- Watchlist : id, userId, videoId, addedAt
- WatchHistory : id, userId, videoId, watchedAt, progressTime, completed
Video Service :
Video : id, title, description, thumbnailUrl, trailerUrl (URL YouTube embed), duration, releaseYear,
type (FILM/SERIE), category (ACTION/COMEDIE/DRAME/SCIENCE_FICTION/THRILLER/HORREUR),
rating, director, cast
Un utilisateur peut avoir plusieurs vidéos dans sa watchlist et son historique
Fonctionnalités par Service
Video Service
• CRUD des contenus vidéo
• Base de données : SGBD Relationnel de votre choix (chaque service doit avoir sa propre
base de données)
User Service
• CRUD des utilisateurs
• Ajouter/Retirer un contenu de la watchlist
• Enregistrer l’historique de visionnage
• Consulter les statistiques de visionnage
• Base de données : SGBD Relationnel de votre choix (chaque service doit avoir sa propre
base de données)
• Communication avec video-service via OpenFeign
Technologies
• Spring Boot
• Spring Data JPA
• Spring Cloud (Eureka, Config, Gateway)
• OpenFeign pour la communication inter-services
• Docker et Docker Compose pour la conteneurisation et l’orchestration des microservices
• L’architecture en couches avec les design patterns (Repository, DTO, Mapper)
• Autres technologies que vous voyez pertinent
Points clés
• Application 100% backend Focus sur les API REST et pas d’interface utilisateur
• Tests exclusivement via Postman
• Trailers YouTube uniquement (pas d’upload de vidéos, seules les URLs embed sont
stockées)
Bonus :
• Ajouter un autre microService réservé pour la sécurité avec une authentification stateless
• Ajouter un autre microService réservé pour le monitoring

Critères de performance
• L’application doit implémenter toutes les fonctionnalités de gestion des utilisateurs et des
contenus vidéo spécifiées pour chaque microservice
• La communication inter-services doit être fiable avec une gestion des erreurs appropriée
• La découverte de services via Eureka doit être opérationnelle
• L’architecture en couches avec les design patterns (Repository, DTO, Mapper) doit être
respectée
• Les tests unitaires doivent couvrir le maximum du code
• Le code doit suivre les bonnes pratiques Java et être documenté
• La configuration centralisée doit être correctement synchronisée avec le repository Git
• Le projet doit être versionné avec Git et géré en méthodologie Agile
